# Eldoria — v1.1 Additions Design Spec

**Date :** 2026-07-02
**Version :** 1.1
**Scope :** 6 fonctionnalités additives sur le thème v1.0.0 déjà implémenté et vérifié en conditions réelles.

---

## 1. Contexte

Le thème Eldoria v1.0.0 est complet (Tasks 1-13 du plan initial) et a été corrigé/vérifié sur une installation Azuriom locale réelle (helpers, routes, vues Shop/Vote reconstruites, support Forum retiré car inexistant chez Azuriom). Cette spec couvre 6 ajouts demandés par le propriétaire du projet, tous vérifiés faisables via inspection du code source Azuriom (`local/azuriom-test/`) avant conception.

Toutes les décisions d'ambiguïté ont été validées avec l'utilisateur :
- i18n : suit la langue globale du site (pas de switcher visible)
- Avatars staff : service par pseudo (minotar.net), pas crafatar (qui exige un UUID)
- Champs staff : pseudo + rôle uniquement
- Hero vidéo : embed YouTube en fond (réutilise `trailer_url`)

## 2. Traduction FR/EN

### Mécanisme (vérifié dans `ThemeManager::loadTheme()`)

```php
if (is_dir($themeLangPath = $this->path('lang'))) {
    trans()->addNamespace('theme', $themeLangPath);
}
```

Azuriom charge automatiquement `<theme>/lang/{locale}/*.php` sous le namespace `theme::` si le dossier existe. La langue effective est `app()->getLocale()`, pilotée par le réglage global **Paramètres > Langue** du site (pas de choix par visiteur — confirmé dans `SettingsRepository::updateConfig()`, cas `'locale' => app()->setLocale(...)`). Azuriom ne supporte que `fr` et `en` en interne (`resources/lang/{fr,en}`) — le thème suit les mêmes codes.

### Fichiers

- Create: `eldoria/lang/fr/theme.php`
- Create: `eldoria/lang/en/theme.php`

Un seul fichier de traduction par langue (fichier `theme.php`, clés organisées par section : `nav`, `footer`, `home`, `shop`, `vote`, `posts`, `staff`, `auth`, `errors`, `customizer`). Structure en tableau PHP imbriqué classique Laravel.

### Modifications

Remplacement de tout texte français codé en dur par `{{ __('theme::theme.section.cle') }}` (ou `@lang(...)` dans les attributs) dans :

`layouts/app.blade.php`, `partials/navbar.blade.php`, `partials/footer.blade.php`, `partials/customizer.blade.php`, `home.blade.php`, `vendor/shop/categories/{index,show,_sidebar}.blade.php`, `vendor/shop/packages/show.blade.php`, `vendor/shop/cart/index.blade.php`, `vendor/vote/index.blade.php`, `auth/{login,register}.blade.php`, `profile/index.blade.php`, `errors/{404,403,500}.blade.php`, plus les nouvelles vues (posts, staff, join-steps) créées dans cette même spec.

**Ce qui reste non traduit (volontairement) :** contenu saisi par l'admin via le customizer (slogan, noms des membres du staff, etc.) — ce sont des données, pas du texte d'interface. L'admin les rédige dans la langue de son choix.

### Vérification

Basculer `locale` entre `fr`/`en` dans les settings Azuriom (via tinker ou DB directe, le panel admin de settings n'étant pas dans le scope de cette spec) et confirmer que les pages du thème changent de langue sans toucher au code.

## 3. Thème des articles / news

### Système réel (vérifié : `app/Http/Controllers/PostController.php`, `app/Models/Post.php`, `resources/views/posts/`)

- Routes : `posts.index` (GET `/news`), `posts.show` (GET `/news/{post:slug}`), `posts.like`/`posts.dislike` (POST/DELETE `/news/{post}/like`), `posts.comments.store`/`.destroy`
- Modèle `Post` : `title`, `description`, `slug`, `content` (HTML), `image` (via `HasImage` : `hasImage()`/`imageUrl()`), `is_pinned`, `published_at`, `author`, `comments`, `likes`
- Vues core actuelles à overrider : `posts/index.blade.php`, `posts/show.blade.php` (chemin racine, PAS `vendor/` — ce sont des vues core, pas un plugin)
- Le cœur d'Azuriom gère les likes/suppression de commentaire via son propre bundle JS (non chargé par notre thème, qui a son propre layout) → à réimplémenter en vanilla JS, même pattern que `vote.js`

### Fichiers

- Create: `eldoria/views/posts/index.blade.php` — grille d'articles (image, titre, extrait tronqué, auteur + date, lien "Lire la suite"), champ de recherche (`?q=`)
- Create: `eldoria/views/posts/show.blade.php` — article complet, bouton like (cœur togglable + compteur), liste de commentaires, formulaire de réponse (si `@can('create', Comment::class)`), message invité sinon
- Create: `eldoria/assets/js/posts.js` — toggle like via `fetch()` POST `posts.like` / DELETE `posts.dislike` (met à jour l'icône + le compteur sans recharger), confirmation native (`window.confirm`) avant suppression d'un commentaire
- Modify: `eldoria/assets/js/app.js` — import + init de `posts.js`
- Modify: `eldoria/views/home.blade.php` — nouvelle section "Actus" (3 derniers `Post::published()->latest('published_at')->take(3)->get()`), même style card que Shop/Vote

### Points d'attention

- `$post->content` est du HTML déjà généré côté admin (éditeur riche) → rendu via `{!! !!}`, pas d'échappement (cohérent avec Forum/Shop déjà traités ainsi pour du contenu admin)
- L'extrait sur `index` utilise `Str::limit(strip_tags($post->content), 250)` comme le fait la vue core (le champ `description` existe mais n'est pas garanti rempli par tous les admins)

## 5. Statut serveur live (AJAX 30s)

### API réelle (vérifiée : `routes/api.php`, `Api\ServerController::index()`, `ServerResource`)

`GET /api/servers` est **public** (aucune authentification, aucune clé) et renvoie :
```json
{
  "default": { "name": "...", "address": "...", "online": true, "players": 12, "max_players": 50 },
  "servers": [ { "name": "...", "online": true, "players": 12, "max_players": 50 }, ... ]
}
```

### Comportement

- Au chargement, puis toutes les 30s : `fetch('/api/servers')`, somme `players` sur tous les serveurs `online`, agrège un booléen "au moins un serveur en ligne"
- Met à jour le compteur "Joueurs en ligne" de la stats-band (remplace l'ancien rendu serveur statique par une valeur vivante)
- Ajoute une pastille verte (en ligne) / rouge (hors ligne) à côté du bouton "Rejoindre" du hero
- **Interaction avec le compteur animé existant** (`initCounters()` dans `animations.js`, qui fait un count-up 0→valeur au premier passage dans le viewport, déclenché sur `data-target`) : le premier fetch au chargement met simplement à jour `data-target` avant que l'animation ne se déclenche (pas de changement visible si le poll est plus rapide que le scroll de l'utilisateur). Les polls suivants (toutes les 30s) écrivent directement le texte du compteur sans rejouer l'animation de comptage — un recomptage depuis 0 toutes les 30s serait un défaut visuel, pas une amélioration.
- Respecte `prefers-reduced-motion` : pas d'animation de transition sur le changement de couleur de la pastille si activé (juste un changement instantané)

### Fichiers

- Create: `eldoria/assets/js/server-status.js`
- Modify: `eldoria/assets/js/app.js` — import + init
- Modify: `eldoria/views/home.blade.php` — attributs `data-live-*` sur le compteur et ajout de la pastille dans le hero

## 6. Section équipe / staff

## 7. Hero vidéo

### Comportement

Nouveau réglage `hero_video_enabled` (bool). Visible uniquement si `trailer_url` est renseigné (sinon pas d'option à cocher, rien à activer). Si actif :
- L'image de fond du hero (`#hero-bg`) est remplacée par un `<iframe>` YouTube en fond (`position: absolute; inset: 0; width: 100%; height: 100%; object-fit: cover`-like via surdimensionnement + `pointer-events-none`), avec `autoplay=1&mute=1&loop=1&controls=0&playlist={id}&modestbranding=1&playsinline=1`
- `loop=1` sur l'API YouTube nécessite `playlist={même_id}` pour boucler sur une seule vidéo (comportement documenté de l'API)
- Le parallaxe GSAP existant sur `#hero-bg` est désactivé si le mode vidéo est actif (un iframe vidéo qui bouge en parallaxe donne un rendu cassé) — géré dans `animations.js` par une vérification de la présence de l'iframe
- Respecte `prefers-reduced-motion` : si activé, le hero retombe sur l'image statique même si `hero_video_enabled` est vrai (une vidéo en autoplay est le cas d'usage principal visé par cette préférence utilisateur)

### Fichiers

- Modify: `eldoria/views/home.blade.php` — rendu conditionnel iframe vs image de fond
- Modify: `eldoria/views/partials/customizer.blade.php` — toggle (visible seulement si `trailerUrl` non vide)
- Modify: `eldoria/assets/js/customizer.js` — état `heroVideoEnabled` + méthode `liveHeroVideo()`
- Modify: `eldoria/assets/js/animations.js` — désactivation du parallaxe si iframe vidéo présente
- Modify: `eldoria/config.json`, `eldoria/config/rules.php` — nouveau champ `hero_video_enabled`

## 8. Ordre des sections sur la homepage

Pour lever toute ambiguïté d'implémentation, l'ordre final des sections de `home.blade.php` après cette spec est :

1. Hero (fond image ou vidéo selon §7, pastille de statut serveur selon §4/§5)
2. Stats band (compteurs live selon §5)
3. Comment nous rejoindre (§4)
4. Trailer (existant, sauf si `hero_video_enabled` — dans ce cas la section Trailer reste affichée normalement, indépendante du fond du hero)
5. Actus / dernières news (§3)
6. Boutique (existant)
7. Vote (existant)
8. Notre équipe (§6)
9. Widget Discord (existant)

## 9. Notes transverses

- **Toutes** les nouvelles sections respectent les contraintes globales déjà en vigueur : mobile-first, `prefers-reduced-motion`, cibles tactiles 48px minimum, couleurs uniquement via les CSS custom properties existantes.
- Le live preview du customizer (déjà en place pour slogan/sections/discord/trailer) est étendu au strict nécessaire (toggle hero vidéo) — le staff n'est pas dans le customizer (voir §6).
- Pas de nouvelle dépendance JS externe : `posts.js` et `server-status.js` utilisent `fetch()` natif, comme `vote.js`.
